## Обзор

Решение тестового задания. Представляет собой API организационной структуры.
Сервис отвечает за работу с подразделениями и их сотрудниками.


## Архитектура

Сервис написан в соответствии с принципами чистой архитектуры.

## Технологии

- **Язык:** python (версия 3.13)
- **API-фреймворк:** FastAPI
- **Фреймворк тестирования:** pytest
- **База данных:** PostgreSQL
- **Миграции:** alembic
- **Контейнеризация:** Docker (+ Compose)

## Основные паттерны

- Clean architecture
- Repository
- Use case + Application service
- Dependency injection

## Особенности реализации

- Чистая архитектура. Ядро: `app/application`. Содержит сущности, интерфейсы,
сервисы. Сценарии с более сложной логикой вынесены в отдельные use cases.
- Версионирование через эндпоинты.
- Взаимодействие с БД абстрагировано Repository-классами.
- Настройка взаимодействия с БД осуществляется через yaml-файл настроек,
  который парсится моделью pydantic-settings.

## Запуск

Для запуска в системе должен быть установлен docker (+ compose). Запуск начнётся 
со сборки docker-образа, если он не был собран ранее. Перед запуском необходимо 
сконфигурировать сервис, для чего требуется создать в корне проекта директорию
`app_config/` со структурой, аналогичной структуре `app_config_example/`.

Запуск:
```bash 
./run.sh
```

Для запуска в тестовом режиме создавать директорию `app_config/` не требуется. 
Сервис запустится с дефолтными (тестовыми) настройками (будет использована
директория `app_config_example/`). Запуск:
```bash
./test_run.sh
```

Swagger-документация: `http://localhost:8000/v1/docs`.

Остановка:
```bash
./stop.sh
./remove_image.sh # Удалить docker-образ
```


## Проблемы/недочёты

Некоторые решения для проекта такого масштаба заведомо избыточны. В особенности
использование чистой архитектуры. В реальном проекте такой подход не всегда
оправдан.

Малое количество тестов.
